3 JThread
4 ---------
5 
6 The heart of the framework is in the JThread objects and the heart of them is in the Loop method.
7 Here is a basic description of what goes on in Loop:
8 
9 The loop is controlled by a member variable "mRunStateTarget". This can be set externally, primarily
10 to tell the loop to pause or quit at the next iteration without interferening with what it is currently
11 doing. Valid values are defined in the RUN_STATE_t enum defined inside the JThread class.
12 
13 The basic workflow in one loop's iteration is:
14 
15 1. Check if events need to be read from the source
16 2. Get the next task from the current event source's JQueueSet and run it if there is one
17 3. If there is no task then call the HandleNullTask method. This will:
18  a. Check if all tasks and sub-tasks from the current source are done and if so,
19  inform the JThreadManager that the current source is exhausted and we need
20  a new one
21  b. If the current source is not completed, but there are currently no tasks available
22  from it for the thread to work on, try switching to another open source and getting
23  tasks from it. (This only happens if multiple sources are open simultaneously.)
24 
25 For item 1. the Loop may actually jump back to the beginning as it reads in events without processing
26 them in an effort to make reading from the source more efficient. The idea is a quick succession of
27 read requests will keep the disk spinning and the head from thrashing so you gain overall by buffering
28 several events at once. How many are buffered is controlled by the the JEventSource (see
29 SetNumEventsToGetAtOnce method.) This feature may not matter for SSD's or reads.
30 
31 A JThread object accesses a JEventSource via the JThreadManager::JEventSourceInfo class.
32 This class is used to group a JEventSource pointer and a JQueueSet pointer together as a
33 single entity. Note that the JEventSource will provide a single JQueue for its events. The
34 JQueueSet is created by the JThreadManager and adds the source's queue as well as any
35 queue's from the base set of queue's in the mTemplateQueueSet member of the JThreadManager.
36 The JQueue objects in the set are clones of the ones found in mTemplateQueueSet so they
37 form a set unique to the JEventSource.

40 JThreadManager
41 --------------------
42 
43 A single JThreadManager object exists for the process. It is created in the JApplication constructor
44 and is owned by the JApplication object. It is responsible for creating and managing threads as
45 well as their associated JQueueSet objects. It also manages which JEventSource object a JThread
46 object uses at any given time.

49 JQueueSet
50 --------------------
51 
52 Each JEventSource will have it's own JQueueSet object created for it. The JQueueSet object is
53 created and owned by the JThreadManager. This is because it must combine the JQueue object
54 from the JEventSource itself with any other JQueue objects coming from reconstruction
55 code. Keeping a separate JQueueSet for each source allows events from that source to be
56 processed independently from other sources. Specifically, barrier events only need to serialize
57 the processing of data from the source they came from. Since JANA supports reading from
58 multiple JEventSource objects simultaneously, unique JQueueSet objects are needed.
59 
60 The JQueueSet is queried for a task from the JThread::Loop. The JQueueSet will loop
61 through it's queue's by type first and then JQueue objects of that type to ensure a priority
62 for the order in which tasks are processed. The JQueueSet keeps the JQueue objects in a
63 std::map with keys being the queue type (defined in an enum that is part of the JQueueSet
64 class). The values of the map are std::vector containers holding the actual queues. Due to the
65 std::map ordering, the priority is given to the queue type in the order they are defined in the
66 enum. Specifically: Output, SubTasks, Events. Thus, if an Event task is performed that creates
67 subtasks, then those will be performed next before another task is pulled from an Event queue.
68 
69 JQueueSet objects are made by combining the JQueue object from a JEventSource and
70 clones of any JQueue objects in the JThreadManager::mTemplateQueueSet object. Plugins
71 and libraries will need to add their JQueue objects to the template queue set by calling
72 the AddQueue method of JThreadManager before event processing starts. Note that
73 the JQueue object passed into AddQueue will never actually be used to hold any tasks.
74 It will just be used to clone itself to make new JQueue objects to go into JQueueSet objects
75 as needed.

78 JEventSource
79 --------------------
80 
81 JEventSource is a base class for classes that read events into the program. User's
82 will need to provide a class that inherits from this in order to provide input events
83 to the JANA program.
84 
85 ## GetEvent:
86 This method will be called to read in one "event" from the source. The
87 event may simply be a large buffer containing one or more physics events. This is
88 called serially so generally very little data processing should be done during this
89 call. See the GetProcessEventTask method for details on how to setup the parsing
90 as a separate that can be done in parallel while other events are being parsed or
91 processed.
92 
93 ## GetObjects:
94 This method will be called when a factory or processor requests some
95 objects for the event. The JEvent passed into this will be of the form that has the
96 objects available. i.e. For more complicated sources that implement multiple
97 queues to hold different forms of the event as it is parsed, this will typically be the
98 final form of the event.
99 
100 ## GetProcessEventTask:
101 This method is called to make a JTask object for a given JEvent.
102 The JEvent will have been obtained by an earlier call to GetEvent(). This is where one
103 would implement code to process the JEvent. For example, the GetEvent method
104 might only read in an event buffer and wrap it as a JEvent object. Any parsing of the
105 buffer is deferred so it can be done in parallel as opposed to parsing it in the GetEvent
106 method which is called serially. The parsing code would naturally fit into the
107 process event task.
108 
109 There is a default GetProcessEventTask method defined in JEventSource that will make
110 a JTask that simply calls all of the JEventProcessors for the JEvent. (This uses the
111 JMakeAnalyzeEventTask routine defined in JFunctions.cc.) If the user provides their
112 own GetProcessEventTask method, they should eventually generate a task that
113 runs the processors.
